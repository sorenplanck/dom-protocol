# DOM RFC-0011 — Bootstrap Discovery, PMMR Peak Bagging & Fee Policy

Status: **Normative**
Depends on: RFC-0000, RFC-0004, RFC-0005, RFC-0008

---

## 1. PMMR Peak Bagging Algorithm (clarifies RFC-0004)

RFC-0004 defines tags and leaf/node hashing but does not specify how peaks are
combined to produce the final PMMR root. This section is the normative definition.

### 1.1 Peak Bagging (Right-to-Left Fold — RFC-0004 canonical)

RFC-0004 already specifies the right-to-left fold:

```
acc = last_peak

for peak in reverse(peaks_without_last):
    acc = Blake2b-256(
        u16_le(len("DOM:pmmr-bag:v1")) ||
        "DOM:pmmr-bag:v1" ||
        peak[32]  ||   ← left peak
        acc[32]        ← right accumulator
    )

root = acc
```

**Order is consensus-critical.** Left peak = peak closer to index 0 (older).
Right accumulator = peak closer to end (newer). Any reversal is consensus-invalid.

### 1.2 Required Test Vectors (per RFC-0004)

For leaf counts 0, 1, 2, 3, 4, 7, 8, 15, 16 where leaf[i] payload = i.to_le_bytes():

```
leaves=0:  empty PMMR root = Blake2b-256("DOM:pmmr-empty:v1" || "")
leaves=1:  leaf_hash(pos=1, b"\x00\x00\x00\x00\x00\x00\x00\x00")
...
```

Reference values MUST be generated by the reference implementation and independently
reproduced before testnet launch.

## 2. Mempool Fee Policy (normative)

### 2.1 Minimum Relay Fee Rate

```
MIN_RELAY_FEE_RATE = 1000 noms/weight_unit  (from RFC-0000, classification: Policy)
```

This is a relay policy — transactions below this rate are rejected by relay nodes
but remain consensus-valid (a block may include them if mined directly).

Response to below-minimum fee: `TemporarilyInvalid("fee below minimum relay rate")`
Do NOT ban the peer — zero-fee transactions are consensus-valid.

### 2.2 Mempool Size and Eviction

```
MAX_MEMPOOL_WEIGHT = 400_000 wu  (10 × MAX_BLOCK_WEIGHT)
EVICTION_POLICY   = lowest-feerate-first
```

When mempool exceeds MAX_MEMPOOL_WEIGHT, evict the lowest-feerate transaction.
If the new transaction has lower fee than the lowest in pool, reject it:
`PolicyRejected("mempool full, insufficient fee to evict lowest entry")`

### 2.3 Zero-Fee Transactions

Zero-fee transactions:
- Are consensus-valid
- Are relay-policy rejected (TemporarilyInvalid, not Invalid)
- May be included by a miner who receives them directly
- Peers that consistently send zero-fee transactions do NOT receive ban score

## 3. Bootstrap Discovery (DNS Seeds and Peer Exchange)

### 3.1 DNS Seeds

Mainnet DNS seeds (operated by independent entities):

```
seed1.dom-protocol.org
seed2.dom-protocol.org
seed3.dom-protocol.org
seed4.dom-protocol.org
seed5.dom-protocol.org
```

**Governance:** Each seed operator publishes the SHA256 of the current IP set,
signed with a key announced in this RFC. At least 3 of 5 operators must agree
before a seed domain is transferred.

### 3.2 Hardcoded Fallback IPs

If DNS resolution fails entirely, nodes fall back to hardcoded IPs.
These are long-running foundation nodes. Specific IPs MUST be frozen before
testnet launch (placeholder — to be filled after genesis).

```
MAINNET_SEED_IPS = []  // to be filled after genesis launch
TESTNET_SEED_IPS = []  // to be filled after testnet launch
```

### 3.3 ADDR Rate Limiting

```
MAX_ADDR_PER_MSG           = 1000
MAX_ADDR_PER_HOUR_PER_PEER = 100
```

Peers exceeding `MAX_ADDR_PER_HOUR_PER_PEER` receive ban score +30 (address flooding).

### 3.4 Minimum Outbound Diversity

```
MIN_OUTBOUND_CONNECTIONS = 8
MAX_PEERS_SAME_SLASH_16  = 2
```

Eclipse protection: no more than 2 peers from the same /16 subnet.
Feeler connections: 2 additional short-lived connections to test new peers.

## 4. RPC API (Minimum for Testnet)

The following endpoints are required for testnet operation. Full RPC spec
will be RFC-0012.

```
GET  /v1/chain/tip              → {height, hash, total_difficulty}
GET  /v1/block/{hash}           → {header, transactions, coinbase}
GET  /v1/tx/{kernel_excess_hex} → {status: "confirmed"|"mempool"|"unknown", height}
POST /v1/tx/broadcast           → submit raw transaction bytes
GET  /v1/outputs/unspent        → paginated UTXO set (page, limit params)
GET  /v1/health                 → {synced, peer_count, mempool_count}
```

Authentication: none required for read endpoints. POST requires local-only bind
or explicit auth token for remote access.

## 5. ProofOfWork Exclusion from Block Weight

For completeness: `BlockHeader` (fixed size) and `ProofOfWork` (nonce + randomx_hash)
are NOT counted in block weight. Block weight is calculated exclusively from:

```
weight(Block) = sum(weight(tx) for tx in non_coinbase_transactions)
```

Where `weight(tx)` uses the formulas in RFC-0010 §1.1.

## 6. RandomX Seed Branch Resolution (normative)

Section "RandomX Seed Schedule" defines `randomx_seed_height(height)` and states
that the seed is the block hash at `randomx_seed_height`. This section resolves
an ambiguity in that phrasing: **which chain** the seed block is taken from when
branches disagree.

**Normative rule:** the RandomX seed for a block is the hash of the block at
`randomx_seed_height(height)` **on that block's own branch** — the ancestor
reached by walking back from the block's parent — NOT the block currently at
that height on the canonical chain.

When a block extends the canonical chain, the two coincide and the canonical
height index MAY be used as an optimization. They diverge only for a side-branch
block validated during a reorg that crosses an epoch boundary
(`RANDOMX_SEED_INTERVAL`); in that case the block's own branch is authoritative.
Using the canonical index there would validate the block against a seed from a
competing branch, splitting consensus — a valid block rejected, or nodes
disagreeing on validity.

A divergence deeper than `MAX_REORG_DEPTH_POLICY` is rejected: such a block is
unpromotable regardless, so the seed resolution refuses rather than falling back
to the canonical seed.
