TASK 01/50 — IMPLEMENT REAL BLOCK-LEVEL CUT-THROUGH VALIDATION

Problem:
The audit identified a critical consensus gap: block-level cut-through exists conceptually but is not enforced as a production validation rule. A block may pass per-transaction validation while violating aggregate block-level semantics.

Objective:
Make block-level cut-through a mandatory part of full block validation, or explicitly remove the rule from the protocol if it is not intended. Prefer implementation.

Tasks:
1. Trace every production block acceptance path: live relay, local mining, IBD, reorg promotion, replay/reopen, and tests.
2. Identify where full block validation is called and where it is bypassed.
3. Integrate cut-through validation into the canonical full-block validation path.
4. Ensure no block can be accepted into canonical chain state without this rule.
5. Ensure the validation result is deterministic and independent of transaction ordering except where the protocol defines canonical ordering.
6. Add adversarial tests:
   - block with individually valid transactions but invalid aggregate cut-through;
   - block with cut-through ambiguity;
   - block accepted through IBD must enforce the same rule as live relay;
   - mined block must satisfy the same rule before broadcast.
7. Add regression coverage in the most appropriate crates, likely dom-consensus and dom-chain.
8. Do not paper over failures by adjusting fixtures unless the fixtures are provably wrong.
