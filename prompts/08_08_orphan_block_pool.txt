TASK 08/50 — IMPLEMENT ORPHAN-BLOCK POOL

Problem:
The audit identified that missing-parent blocks are mostly dropped as Orphan and live convergence depends on retransmission luck and message ordering.

Objective:
Implement a deterministic orphan-block subsystem so valid children delivered before parents are retained and reprocessed.

Tasks:
1. Add an orphan pool keyed by parent_hash and block_hash.
2. Define strict bounds:
   - global max orphan count;
   - per-peer max orphan count;
   - max block bytes;
   - max orphan depth if applicable.
3. When a child arrives before its parent, store it if valid enough for orphan admission.
4. When the parent arrives, reprocess ready orphans in canonical order.
5. Canonical order must not depend on HashMap iteration.
6. Add peer abuse controls without permanently banning configured bootstrap peers for benign reorder.
7. Add tests:
   - child before parent converges;
   - multiple orphans for same parent process deterministically;
   - orphan spam is bounded;
   - restart policy is explicit and tested.

Validation commands:
- cargo fmt
- cargo check
- Run the narrow relevant cargo test commands for the touched crates.
- If this touches integration behavior, run cargo test -p dom-integration-tests -- --test-threads=1.
