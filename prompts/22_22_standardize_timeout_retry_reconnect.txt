TASK 22/50 — STANDARDIZE TIMEOUT/RETRY/RECONNECT POLICY

Problem:
Inconsistent retry and reconnect behavior can cause unstable P2P, duplicate loops, and scheduler-sensitive convergence.

Objective:
Implement a clear bounded retry/reconnect policy.

Tasks:
1. Define shared retry/backoff configuration:
   - initial delay;
   - max delay;
   - jitter policy;
   - reset after stable session;
   - max in-flight attempts.
2. Apply to P2P outbound connections and backbone/bootstrap peers.
3. Prevent duplicate reconnect loops for the same peer.
4. Ensure configured bootstrap peers are not permanently poisoned by temporary failures.
5. Separate operational retry from consensus decisions.
6. Add tests:
   - backoff increases;
   - stable session resets backoff;
   - duplicate reconnect loop is prevented;
   - configured peer remains eligible after failure.
